## Item 1: Non-Code Project Support

### What
Add a `type` field to the project data model. Two values: `"code"` (default) and `"config"`. Config projects skip git validation but keep the full stage-gate workflow.

### Where to Change

**`src/projects.js`:**
- `addProject()` — accept `type` parameter, default to `"code"`, store in project object
- Any function that validates git (branch detection, commit history, repo checks) — add `type === "config"` bypass
- `resolveProjectPath()` — no change needed (works for both types)

**`data/projects.json`:**
- Existing projects do NOT get a `type` field added automatically. They continue to work as `"code"` (the default). Backward compatible — if `type` is undefined, treat as `"code"`.

**`server.js`:**
- `POST /api/projects` — pass `type` through to `addProject()`
- `PUT /api/projects/:id` — allow updating `type`
- `POST /api/projects/:id/evaluated` — skip git checks when `type === "config"`
- `POST /api/scaffold-project` — pass `type` through
- Any route that does git validation — add config bypass

**`public/app.js`:**
- Project creation form/modal — add a project type selector (dropdown or toggle: Code / Config). Default: Code.
- Project edit modal — show type field, allow changing it
- Project card display — optionally show a small indicator for config projects (subtle, not loud)

### What NOT to Change
- Stage-gate workflow — identical for both types
- Treeview — works the same regardless of type
- Kickoff prompt detection — same for both types
- Test file generation — same for both types

### Acceptance Criteria
- [ ] Create a new project with type "config" — it can be activated without git
- [ ] Create a new project with type "code" — existing behaviour, git required
- [ ] Existing projects (no type field) — behave as "code" (backward compatible)
- [ ] Config project has full stage-gate workflow (stages, kickoff prompts, test files)

## Item 2: Development Workflow Enforcement + Slash Command Alignment

### What
Verify and enforce that CCC source code respects the Cowork→CC pipeline. CC cannot auto-generate tasklists or stage plans. These are Cowork's job. Additionally, the slash command templates that CCC deploys to scaffolded/imported projects must match the corrected workflow.

### Part A: Code Audit

**`server.js` — scaffold routes:**
- `POST /api/scaffold-project` (line ~910) — check if this creates a tasklist automatically. If it does, REMOVE the tasklist creation. Scaffolding should create the project folder structure (docs/, docs/handoff/, etc.) but NOT generate a tasklist with tasks. An empty tasklist template is OK.
- `POST /api/scaffold-import` (line ~1043) — same check. If it generates a tasklist, remove that logic.

**`src/projects.js`:**
- Search for any function that creates, generates, or populates a tasklist from a concept doc. Remove or gate it.
- Search for any function that creates stage plans or task breakdowns. Remove or gate it.

### Part B: Slash Command Templates in `generateSlashCommand()`

**Problem:** `generateSlashCommand()` in `server.js` (~line 846) only knows three commands (`update-tasklist`, `review-concept`, `status`) and their instruction text is outdated — e.g. `update-tasklist` still says "Add any new tasks that have emerged during implementation" which directly violates the workflow.

Meanwhile, CCC's own `.claude/commands/` directory has eight corrected command files. The templates must match.

**Fix:** Rewrite `generateSlashCommand()` to support all eight commands with the corrected instruction text. Use CCC's own `.claude/commands/` files as the reference — they contain the approved workflow-aligned instructions.

The full command set to deploy:

1. **`start-stage`** — Reads kickoff prompt from `docs/handoff/stage{XX}-prompt.md`, executes it. If no kickoff prompt exists, STOP and inform developer. Does NOT interpret concept doc. Does NOT create its own plan. The kickoff prompt IS the approval — no asking "shall I proceed?"

2. **`continue`** — Resume from where last session stopped. Reads CLAUDE.md, tasklist, checks for kickoff prompt (refuses without one), checks git log, brief status then works. Same autonomy rules as start-stage.

3. **`update-tasklist`** — Mark tasks complete in CURRENT STAGE ONLY. Do NOT add new tasks, do NOT remove tasks, do NOT modify future stages, do NOT update stage status. The tasklist is owned by Cowork. CC only updates completion checkboxes.

4. **`review-concept`** — Compare kickoff prompt (primary) and concept doc (background) against implementation. Flag drift, scope creep, unaddressed items. Drift measured against kickoff prompt, not concept doc.

5. **`status`** — Read tasklist, count done/remaining in current stage, list blockers. Short format.

6. **`create-tasklist`** — DEPRECATED guard. Output: "Tasklists are created by Cowork. CC does not create tasklists. If no tasklist exists, inform the developer." Do NOT generate anything.

7. **`eod`** — End-of-day wrap-up. Git log for today, tasklist progress, uncommitted changes, summary (done/in-progress/blockers/tomorrow).

8. **`test`** — Run test suite, build, lint. Report pass/fail counts, failures with file+line, build status.

### Part C: Update Scaffold Deployment

**Both scaffold routes** (`POST /api/scaffold-project` and `POST /api/scaffold-import`) currently deploy only three commands:
```javascript
const slashCommands = ['update-tasklist', 'review-concept', 'status'];
```

**Fix:** Update both routes to deploy all eight commands:
```javascript
const slashCommands = ['start-stage', 'continue', 'update-tasklist', 'review-concept', 'status', 'create-tasklist', 'eod', 'test'];
```

### Part D: Remove `/start-project` UI references

The import modal shows notices referencing `/start-project` (app.js ~line 2657, 2660):
- "No CLAUDE.md detected. It will be generated when you run `/start-project`."
- "No tasklist detected. It will be generated when you run `/start-project`."

There is no `/start-project` command. These notices are misleading. Fix:
- CLAUDE.md notice: "No CLAUDE.md detected. It will be scaffolded during import."
- Tasklist notice: "No tasklist detected. The tasklist will be created in a Cowork session after import."

### What NOT to Change
- The stage-gate workflow itself — just who controls it
- The tasklist file format — stays the same
- The concept doc format — stays the same
- CCC's own `.claude/commands/` files — they're already correct, use them as reference only

### Acceptance Criteria
- [ ] `POST /api/scaffold-project` does NOT create a populated tasklist (empty template is OK)
- [ ] No code path in CCC auto-generates tasks from a concept doc
- [ ] `generateSlashCommand()` returns correct workflow-aligned text for all 8 commands
- [ ] Both scaffold routes deploy all 8 slash commands to new projects
- [ ] `/update-tasklist` template says "Do NOT add new tasks" — not "Add any new tasks"
- [ ] `/start-stage` template refuses without kickoff prompt
- [ ] `/create-tasklist` template is a deprecation guard, not a generator
- [ ] No UI references to `/start-project` remain
- [ ] Existing projects are not affected (their `.claude/commands/` files are NOT overwritten)

## Item 3: Test File Version-Aware Placement

### What
Test files must be placed in the correct version subfolder based on the project's active version.

### Logic
```
activeVersion = project.activeVersion (from projects.json / project data)

if activeVersion is three-part (e.g., "1.0.6"):
  majorMinor = "v1.0"
  full = "v1.0.6"
  path = docs/{majorMinor}/{full}/{ProjectName}_test_stage{XX}.md

if activeVersion is two-part (e.g., "1.0"):
  path = docs/v{activeVersion}/{ProjectName}_test_stage{XX}.md
```

### Where to Change

**Wherever test file paths are constructed:**
- Search the entire codebase for `_test_stage` — this is the regex pattern. Find every place that constructs or references a test file path.
- The CLAUDE.md test file generation rule already documents this — but the code must enforce it.
- If there's a helper function that builds test file paths, update it. If paths are constructed inline, create a helper.

**Helper function (suggested):**
```javascript
function getTestFilePath(projectName, stageNumber, activeVersion) {
  const parts = activeVersion.split('.');
  if (parts.length === 3) {
    // Patch version: docs/v{X.Y}/v{X.Y.Z}/
    const majorMinor = `v${parts[0]}.${parts[1]}`;
    const full = `v${activeVersion}`;
    return `docs/${majorMinor}/${full}/${projectName}_test_stage${String(stageNumber).padStart(2, '0')}.md`;
  } else {
    // Major version: docs/v{X.Y}/
    return `docs/v${activeVersion}/${projectName}_test_stage${String(stageNumber).padStart(2, '0')}.md`;
  }
}
```

**Auto-create subfolder:**
- When constructing the test file path, create the directory if it doesn't exist (`fs.mkdirSync(dir, { recursive: true })`)

**Treeview:**
- The treeview scans for test files using regex `/_test_stage\d+\.md$/`. This matches regardless of folder depth — verify it scans recursively into version subfolders. If it only scans one level deep, fix the scan depth.

### What NOT to Change
- Test file naming pattern — only the directory changes
- Test file content format — unchanged
- Who generates test files — CC still generates them (per ADR-019)

### Acceptance Criteria
- [ ] Project with activeVersion "1.0.6" — test file created at `docs/v1.0/v1.0.6/{name}_test_stageXX.md`
- [ ] Project with activeVersion "1.0" — test file created at `docs/v1.0/{name}_test_stageXX.md`
- [ ] Version subfolder auto-created if it doesn't exist
- [ ] Treeview displays test files from both locations

## Item 4: Project Activation Fix

### What
Projects created in CCC (via scaffold), imported, or initialized cannot always be dragged from Parked to Active. The `evaluated` field gates the move, but projects get `evaluated: undefined` when created through certain paths, which blocks them. An ad-hoc fix on 2026-03-27 changed `evaluated !== true` to `=== false` — this helped but only patched one symptom. Replace with a proper solution.

### Root Cause
The `evaluated` field was designed to flag imported projects as "needs attention" (`evaluated: false`). But the activation checks treat anything that isn't explicitly `true` as blocked. Projects created via CCC UI or imported get `evaluated: undefined` initially, which fails the checks.

The startup migration (`startupMigration()` in `server.js` ~line 1434) backfills `evaluated: true` for projects where it's `undefined` — but this only runs on server start, not when projects are created mid-session.

### Where to Change

**`public/app.js` — `handleDrop()` (~line 1245):**
Current code blocks drag to Active when `evaluated === false`. This is the RIGHT check — unevaluated imports should be blocked. BUT: the check must also handle `config` type projects. Config projects skip evaluation entirely.

```javascript
// Current (ad-hoc fix):
if (targetGroup === 'Active' && dragged.evaluated === false) {
  showWarning('Run /evaluate-import before moving this project to Active.');
  ...
}

// Fix: config projects bypass the check entirely
if (targetGroup === 'Active' && dragged.evaluated === false && dragged.type !== 'config') {
  showWarning('Run /evaluate-import before moving this project to Active.');
  ...
}
```

**`public/app.js` — edit modal validation (~line 2903):**
Same pattern — blocks moving to Active when `evaluated === false`. Add config bypass:

```javascript
// Current:
if (group === 'Active' && project.evaluated === false) {
  ...
}

// Fix:
if (group === 'Active' && project.evaluated === false && project.type !== 'config') {
  ...
}
```

**`public/app.js` — `startSession()` guard (~line 464):**
Current logic: `proj.group !== 'Active' && proj.evaluated !== false`. This blocks Parked+evaluated projects from starting sessions — that's intentional (must move to Active first). But also blocks Parked+undefined. The fix: this is correct as-is after the startup migration backfills `undefined` → `true`. Verify this logic is sound.

**`public/app.js` — terminal prompt area (~line 1367):**
Same pattern as `startSession()`. Shows "Move to Active" notice for parked+evaluated projects. Verify consistent with the above.

**`public/app.js` — import flow (~line 2854):**
Sets `evaluated: false` for imports. This is CORRECT — imports need evaluation. No change needed.

**`server.js` — `POST /api/scaffold-project` (~line 1028):**
Sets `evaluated: true` for scaffolded projects. This is CORRECT — scaffolded projects have all docs. No change needed.

**`server.js` — startup migration (~line 1452):**
Backfills `evaluated: true` for `undefined`. This is CORRECT but only runs on startup. Add the same backfill to `addProject()` in `projects.js` so new projects always get an explicit `evaluated` value:

```javascript
// In addProject() — ensure evaluated is always explicitly set
if (projectData.evaluated === undefined) {
  projectData.evaluated = true;
}
```

**`src/projects.js` — `updateProject()` allowed fields (~line 102):**
Currently allows: `['name', 'group', 'coreFiles', 'activeVersion', 'evaluated']`. Add `'type'` to this list (needed for Item 1 as well).

### Summary of Changes
1. `handleDrop()` — add `&& dragged.type !== 'config'` to evaluation check
2. Edit modal — add `&& project.type !== 'config'` to evaluation check
3. `addProject()` in `projects.js` — default `evaluated: true` if not explicitly set
4. `updateProject()` allowed fields — add `'type'`
5. Verify `startSession()` and terminal prompt logic remain correct

### What NOT to Change
- The `evaluated: false` flag for imports — that's intentional
- The startup migration — it's correct
- The `/evaluate-import` command flow — unchanged

### Acceptance Criteria
- [ ] Create a project in CCC → drag from Parked to Active → succeeds
- [ ] Import a project → drag from Parked to Active → blocked (correct — needs evaluation)
- [ ] Import a project → run `/evaluate-import` → drag to Active → succeeds
- [ ] Config project → drag from Parked to Active → succeeds (skips evaluation check)
- [ ] Existing projects with `evaluated: true` → behaviour unchanged
